PART 1 — 앱 유저 흐름
1
유저

이벤트 티켓 획득 — 출석 5일 연속 달성 보상

이벤트 티켓은 경품 응모 전용 티켓으로, 구매 적립 티켓(랜덤/등급)과 완전히 분리된다. 획득 경로는 출석 5일 연속·행운룰렛·온보딩 가입(1회성) 등이며, 전 경로 합산 일 5장 / 월 30장 한도가 적용된다. 화면은 5일 연속 출석 달성 시 이벤트 티켓 1장이 지급되는 보상 화면 — 티켓 카드에 "경품 응모 전용" 배지가 붙고, 하단 "경품 응모 바로가기" 버튼으로 응모 화면과 직접 연결된다.

출석 5일 달성 보상 · 이벤트 티켓 +1장 지급 · "경품 응모 바로가기" 연결
불일치 — 이 화면의 티켓 문구가 "유효기간 30일"(2곳)로 표시됨. 정책 최종안(마스터 정책서)은 발급일+100일 고정 소멸. 하단 미결·불일치 (2)번 참조.
2
유저

혜택 탭 — 경품 응모 진입 카드

혜택 탭 중단의 "이번 주 행운의 추첨" 카드가 경품 응모 화면의 진입점이다. 카드에 보유 이벤트 티켓 수와 이번 회차 경품 요약이 노출된다. 그 아래 "진행중 이벤트" 리스트에도 주간 추첨 이벤트 행이 있다.

혜택 탭 · "이번 주 행운의 추첨" 카드 + 진행중 이벤트 리스트
불일치 — 진행중 이벤트의 "주간 추첨 이벤트" 행 설명이 "전주 접속 유저 자동 응모"로 표시됨. 응모 화면(다음 단계)과 정책 최종안은 티켓을 직접 소모하는 수동 응모. 하단 미결·불일치 (3)번 참조.
3
유저 정적 목업 · 실연동 미완

경품 응모 메인 — 회차 정보 확인 · 응모

상단 히어로 카드에 이번 회차 경품 구성(등수별 경품·당첨자 수)과 마감 D-day가 노출되고, "이벤트 티켓 1장으로 응모하기" 버튼으로 응모한다. 그 아래 "내 응모 현황"에서 이번 주 응모 횟수·남은 티켓·누적 응모를 확인한다. 응모하면 티켓이 즉시 소모되며 취소할 수 없다(1단계 기준). 1회 추첨에 누적 최대 10장까지 응모 가능하고, 응모 티켓 수에 비례해 당첨 확률이 올라간다.

경품 응모 메인 상단 · 회차 경품 구성 + 마감 D-day + 응모 버튼 + 내 응모 현황
구현 상태 — 이 화면은 현재 데모에서 정적 목업이다. 표시값(D-3, 보유 3장, 응모 3회, 누적 17회, 경품 구성)이 전부 하드코딩이고, 응모 버튼을 눌러도 실제 응모 없이 혜택 탭으로 돌아간다. CMS 추첨 데이터와도 연결되어 있지 않다.
4
유저

티켓 더 모으기 · 지난 회차 당첨 결과

응모 화면 하단에서 이벤트 티켓 획득 경로 3종(출석 5일·행운룰렛·온보딩 가입)의 오늘 상태를 확인하고 각 미션으로 이동할 수 있다. 맨 아래 "지난 회차 당첨 결과"는 직전 회차의 등수별 경품과 당첨자(마스킹 닉네임·인원)를 보여준다.

경품 응모 메인 하단 · 티켓 획득 경로 3종 + 지난 회차 당첨 결과
PART 2 — CMS 운영자 흐름
5
운영자

추첨 관리 — 회차 목록 · 당첨 처리 대기

CMS "추첨 관리"에서 추첨 회차를 등록·운영한다. 회차별로 경품 구성·추첨 기간·마감일·참여 티켓·당첨자·상태를 관리하고, 마감됐지만 당첨 처리(추첨 실행)를 안 한 회차 수가 좌측 메뉴 배지로 표시된다. 정책 최종안 기준 1단계 추첨 실행은 관리자 수동 — 당첨자에게 카톡/이메일로 배송지 입력을 안내하고 경품을 수동 발송한다.

CMS > 추첨 관리 · 회차 목록 (현재 등록된 회차 없음 — 빈 상태)
구현 상태 — 회차 목록은 빈 상태인데 좌측 메뉴 배지는 "1"로 표시됨(배지 값 하드코딩). 화면 부제 "추첨 방식: 매주 월요일 자동 발표(자동/수동 전환 가능)"는 정책 최종안(관리자 수동·주기 미결)과 불일치. 하단 미결·불일치 (1)(5)번 참조.

운영자

새 추첨 회차 등록

회차명·시작일·마감일을 입력하고 등수별 경품 구성을 편성한다(등수당 1행·중복 등수 금지, 한 등수에 경품 여러 개 가능, 등수별 당첨자 수 지정). 총 참여 티켓 소모는 자동 계산 항목이다.

CMS > 추첨 관리 > 새 추첨 회차 등록 · 기간 + 등수별 경품·당첨자 수 편성
7
운영자 모델 불일치

경품/응모 관리 — 응모 집계 · 경품 설정 · 발송 관리

오늘 응모 수·오늘 당첨자·소모 이벤트 티켓·미발송 경품 KPI를 집계하고, 현재 경품 설정을 편집한다. 당첨됐지만 아직 발송 안 한 경품은 "미발송 경품" 목록에서 처리하고, "경품 변경 이력"으로 경품 구성 변경을 버전·공표 여부와 함께 추적한다(누가·언제·전후·공표).

CMS > 경품/응모 관리 · KPI 4종 + 현재 경품 설정 + 미발송 경품 + 변경 이력
불일치 — 이 화면의 "현재 경품 설정"은 등수별 당첨 확률 %(1등 5%·2등 15%·3등 꽝 80%) 구조, 즉 즉석 확률형 모델이다. 정책 최종안(회차 모집 후 응모 티켓 수 비례 비복원 추첨)·앱 응모 화면(회차형)·추첨 관리의 회차 등록 구조와 모델 자체가 다르다. 하단 미결·불일치 (4)(6)번 참조.
8
운영자 모델 불일치

주간 이벤트 추첨 — 자동 응모 방식 설정 화면

이벤트 티켓 보유 회원을 매 회차 자동 응모시키고 매주 월요일 00:00 자동 추첨하는 방식의 설정 화면. 응모 대상·추첨 일정·당첨 확률(50%, 조정 가능)·당첨 보상(경품 지급, 티켓 신규 발급 없음)·이벤트 티켓 만료(발급일+100일)를 정의하고, 이번주 자동 응모자·소진 티켓을 집계한다.

CMS > 주간 이벤트 추첨 · 추첨 설정 5항목 + 이번주 추첨 현황
불일치 — 이 화면은 "보유 티켓 자동 응모 + 당첨 확률 50% + 매주 자동 추첨" 모델로, 정책 최종안(직접 응모·티켓 수 비례 확률·관리자 수동 추첨)과 응모 방식·확률·실행 방식이 모두 다르다. 같은 CMS 안의 "경품/응모 관리"(직접 응모 전제)와도 충돌. 티켓 만료 100일만 정책과 일치. 하단 미결·불일치 (3)(4)(5)번 참조.
PART 3 — 정책 · 규칙 (마스터 정책서 확정 기준)

확정 정책 — 출처: 마스터 정책서(policyspec.html) "추첨·경품" 및 "이벤트 티켓" 항목

응모 수단
이벤트 티켓 전용. 경품 응모에만 사용 가능하고 기프티콘 교환에는 쓸 수 없다. 구매 적립 티켓(랜덤/등급)과 완전 분리.
응모 규칙
응모 즉시 티켓 소모, 취소 불가(1단계 기준). 1회 추첨 누적 최대 10장 응모.
당첨 확률
응모한 티켓 수에 비례해 상승. 당첨자 수만큼 비복원 추첨(한 번 뽑힌 응모는 다시 안 뽑힘).
추첨 운영
1단계는 관리자가 직접 추첨 실행. 당첨자에게 카톡/이메일로 배송지 입력 안내 후 수동 발송.
경품 예산
매출의 일정 비율로 고정 — 티켓 발행량과 무관하게 손실 상한 보장. (비율 %는 미결)
티켓 획득
추첨은 이벤트 티켓을 소비만 하고 새로 발급하지 않는다. 획득 경로는 출석 5일 연속·행운룰렛·온보딩 가입 등 별도 경로. 전 경로 합산 일 5장 / 월 30장 한도(초과 시 0장).
티켓 만료
받은 날부터 100일 뒤 소멸(고정)·이월 없음. 미응모해도 다음 회차로 넘어가지 않고 만료.
알림 연계
경품 응모·당첨 알림은 "이벤트·티켓 알림" 그룹에 포함(티켓 만료 임박 D-30/D-7 알림 포함).

미결 정책 — 형님(운영) 확정 대기

마스터 정책서에 미결로 명시된 항목. 이 문서는 임의로 확정하지 않고 그대로 표시한다.

추첨 주기
월 1회 vs 격주 미확정. 정책서 추천: 월 1회(예산 예측·운영 부담 최소). 현재 앱·CMS 화면 샘플은 "매주"로 표기되어 있어 확정 시 전면 정합 필요.
경품 예산 비율
매출 대비 % 미확정. 정책서 추천: 매출의 1~2%(손실 상한 보장).

  • 정책 기준은 전부 마스터 정책서(policyspec.html — PricePick 정책 SoT)의 "추첨·경품"·"이벤트 티켓" 항목이며, 미결 항목은 임의 확정 없이 미결로 표시했다.
  • 이벤트 티켓 획득 경로의 상세(출석·행운룰렛 규칙)는 각 기능 스토리보드(출석체크·행운 룰렛)에서 다룬다. 이 문서는 "경품 응모 전용 소비처" 관점만 담는다.
