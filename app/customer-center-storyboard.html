고객센터 · 1:1 문의 스토리보드

문의 접수부터 운영자 답변·CMS 집계까지 — 실제 배포본 화면 캡처 기반

이 문서의 모든 화면은 배포본 공개 URL(/app 게스트 · /cms firstadmin)에 헤드리스 브라우저로 직접 접속해 실행·캡처한 스크린샷입니다. 단, 현재 배포본은 앱 접수 버튼이 서버 저장을 하지 않는 미구현 상태라(하단 미결 1번), "처리 중 → 답변 완료" 상태 화면 검증에 필요한 문의 데이터 2건은 콘솔에서 실제 Firestore에 생성하고 관리자 권한으로 답변을 저장한 뒤, 앱·CMS의 실제 렌더링 로직이 그려낸 화면을 캡처했습니다. 그림으로 그려낸 화면은 없습니다.

1
유저

진입 — 마이 > 도움말 > 고객센터 · 1:1 문의

고객센터는 마이 탭의 "도움말" 그룹에서 진입한다. 같은 그룹에 공지사항, 이용약관·개인정보처리방침이 함께 배치되어 있다. 이 밖에 계정 차단 안내(error-blocked)·닉네임 충돌 등 오류 화면의 "고객센터 문의" CTA로도 연결된다.

마이 탭 · 도움말 그룹 (고객센터·공지사항·약관)
2
유저

고객센터 홈 — 상담 채널 2개 + 내 문의 내역

상단 배너에 운영 정보 3종(운영 시간 평일 10–18시 / 평균 답변 1–2 영업일 / 주말·공휴일 휴무)을 고정 노출한다. 상담 채널은 카카오채널 상담(실시간 채팅)과 1:1 문의(비동기 접수) 두 가지. 하단에 내 문의 내역이 붙고, "문의 내역은 탈퇴 전까지 영구 보존됩니다" 문구를 노출한다.

고객센터 홈 · 첫 진입(문의 없음) 빈 상태
3
유저

채널 A — 카카오채널 상담 (실시간)

"카카오채널 상담"을 누르면 브릿지 화면을 거쳐 외부 카카오톡 채널로 연결된다. 앱은 채널 URL 딥링크를 여는 역할만 하고, 실제 상담 응대는 카카오채널에서 이뤄진다 — 앱이 별도로 처리할 로직 없음 (DevQA #8 ① 확정 답변, 2026-07-06).

카카오채널 연결 브릿지 (데모: 시뮬레이션, 실채널 URL 미연동)
전제: PricePick 공식 카카오채널을 운영. 실서비스에서는 채널 URL만 연동하면 된다. 데모는 브릿지 연출 후 "돌아가기"만 제공.
4
유저

채널 B — 1:1 문의 작성

문의 유형(5종 중 택1) + 제목(최대 50자) + 내용(최대 500자)을 입력한다. 세 항목이 모두 채워져야 "접수하기"가 활성화된다. 폼 하단에 답변 운영 시간 안내를 고정 노출.

초기 상태 · 접수하기 비활성
유형 5종: 티켓 적립 / 기프티콘 교환 / 티켓 교환·환수 / 계정·로그인 / 기타
작성 완료 · 접수하기 활성 (글자수 카운터 72/500)
내용 입력란 플레이스홀더는 "최소 10자 이상"이지만 실제 활성화 조건은 1자 이상 — 검증 로직 불일치 (미결 4번).
5
시스템

접수 완료 — 토스트 안내 + 중복 접수 방지

접수하기를 누르면 버튼이 즉시 비활성화(중복 접수 방지)되고 "문의가 접수되었습니다. 평일 1–2일 내 답변드릴게요." 토스트가 노출된다. 접수된 문의는 inquiries 컬렉션에 상태 pending(미처리)으로 쌓이고 CMS 집계에 잡힌다.

접수 토스트 · 버튼 비활성 전환
현재 배포본은 이 버튼이 토스트만 띄우고 Firestore 저장을 하지 않는다 (미결 1번). 아래 6단계부터의 데이터 흐름은 문서 캡처를 위해 동일 스키마로 직접 생성한 실데이터 기준.
6
유저

내 문의 내역 — "처리 중" 상태

접수된 문의는 고객센터 홈의 "내 문의 내역"에 최신순으로 쌓인다. 카드마다 유형 뱃지 + 상태 뱃지(처리 중 / 답변 완료) + 제목 + 접수일을 표시. 답변 전에는 주황 "처리 중"으로 노출된다.

내 문의 내역 · 2건 모두 "처리 중" (티켓 적립 / 계정)
7
운영자 · CMS

CMS 집계 — 대시보드 KPI + 1:1 문의 관리

대시보드에 "미처리 문의" KPI(pending 상태 실집계)가 뜨고, 좌측 네비 "1:1 문의" 메뉴에 미처리 건수 배지가 붙는다. 문의 관리 화면은 통계 4카드(미처리 / 처리 중 / 완료 / 평균 응답 시간) + 검색·상태·유형 필터 + 문의 목록(제목·상태·회원 3행 식별·접수일·유형)으로 구성. 상단 안내: 완료 건도 재오픈 가능하며 재오픈 시 변경 사유·이력(누가·언제)을 기록한다.

CMS 대시보드 · "미처리 문의 2 (pending 상태)" 실집계
CMS > 운영 > 1:1 문의 · 통계 카드 + 목록 (좌측 네비에 배지 2) — 유저가 접수한 문의가 닉네임·연동 상태·식별자와 함께 집계됨
8
운영자 · CMS

문의 상세 / 답변 작성

목록에서 문의를 클릭하면 상세 모달이 열린다. 문의 원문 확인 → 답변 작성 → 처리 상태 지정(완료 등) → "답변 발송". 권한 정책상 CS 역할은 조회 + 1:1 문의 답변만 가능하고 다른 데이터 수정은 불가.

문의 상세 / 답변 모달 · 답변 작성 + 처리 상태 선택 + 답변 발송
현재 배포본의 이 모달은 정적 목업이다 — 클릭한 문의와 무관한 고정 예시 내용이 뜨고(캡처의 메타 "— · — · —"), "답변 발송"도 토스트만 띄운다 (미결 2번).
9
답변 완료

답변 저장 → CMS "완료" 집계 · 앱 "답변 완료" 반영

답변이 저장되면(상태 answered + 답변 본문 + 답변 시각) CMS 통계가 미처리 1 / 완료 1로 갱신되고 평균 응답 시간이 계산된다. 유저 앱에서는 해당 문의가 초록 "답변 완료" 뱃지로 바뀌고, 카드 안에 관리자 답변이 인라인으로 표시된다. 별도 알림(푸시) 없이 문의 내역 재진입 시 확인하는 구조.

CMS · 답변 저장 후 통계 갱신 (미처리 1 / 완료 1 · 평균 응답 시간 산출)
앱 · "답변 완료" 뱃지 + 관리자 답변 인라인 노출

미결 · 불일치 — 형님 확인 필요

임의로 확정하지 않고 표시만 해둔 항목. 1·2번은 데모 실연동 작업, 3~7번은 정책·문구 결정 사안.

  1. 앱 "접수하기" 미구현(깡통) — 토스트만 띄우고 inquiries에 저장하지 않는다. 유저가 문의를 보내도 CMS에 안 잡힘. 실연동 필요 (보안규칙은 이미 로그인 유저 create 허용으로 준비돼 있음).
  2. CMS 답변 모달 미구현(정적 목업) — 어떤 문의를 클릭해도 고정 예시가 뜨고 "답변 발송"은 토스트만. 클릭 문의 바인딩 + 답변 저장(answer·status·answered_at) 실연동 필요. 재오픈 이력 기록도 미구현.
  3. 문의 유형 체계 3곳 불일치 — 앱 작성(티켓 적립/기프티콘 교환/티켓 교환·환수/계정·로그인/기타) vs 앱 내역 렌더(ticket/gifticon/postback/account/point) vs CMS 라벨·필터(postback/account/gifticon/general · 티켓/적립/계정/기타). 실제로 CMS 목록에 미등록 유형이 ticket 원문 그대로 노출됨(7단계 캡처에서 실증). 유형 코드 단일화 필요.
  4. 내용 최소 길이 검증 불일치 — 플레이스홀더 "최소 10자 이상" vs 실제 활성 조건 1자 이상.
  5. "최근 3개월" 라벨 vs 실동작 — 앱 목록은 기간 필터 없이 전체 조회. "영구 보존" 문구와 함께 기간 정책(표시 범위) 확정 필요.
  6. 고객센터 정책이 마스터 정책서(SoT) 미수록 — policyspec에는 CS 수동 지급·영수증 인증만 있고 운영 시간·답변 SLA·유형·보존·재오픈 등 고객센터 규정 자체가 없다. 현재는 화면 문구가 사실상의 정책. 이 스토리보드 확정 시 SoT 반영 권장.
  7. 카카오채널 실채널 URL 미연동 — 브릿지는 연출만. 실서비스 전 공식 채널 개설 + URL 연동 필요.

참고 사항

  • 모든 화면은 배포본 공개 URL에서 캡처: /app은 신규 게스트 계정으로 온보딩부터 실행, /cms는 firstadmin 계정으로 로그인해 실집계 화면을 확인했다.
  • 앱 접수 버튼이 미구현(미결 1)이라, 상태 흐름 검증용 문의 2건은 동일 스키마(user_id·category·title·content·status·created_at)로 Firestore에 직접 생성했고, 답변은 관리자 권한으로 answer·answered_at·status:answered를 저장했다. 화면에 보이는 목록·뱃지·집계·인라인 답변은 전부 배포본의 실제 렌더링 로직이 그린 결과다.
  • 캡처에서 앱 좌측의 KO/VI 언어 전환 플로팅 버튼은 화면 가림 방지를 위해 숨김 처리했다 (앱 기능과 무관한 데모용 오버레이).
  • 문서 캡처용으로 생성한 문의 2건은 inquiries 컬렉션에 남아 있다 (데이터 관리(데모) 초기화 시 함께 삭제됨).
