온보딩

알림 권한 프리퍼미션 → iOS 시스템 팝업

스플래시 직후 앱 자체 프리퍼미션 화면이 먼저 뜬다. 혜택(적립·등급 전환·만료 임박 D-30/D-7·이벤트 소식)을 설명한 뒤 '다음'을 누르면 그때 iOS 시스템 권한 팝업을 노출한다. '나중에 설정하기'로 건너뛸 수 있으며, 허용/거부와 무관하게 닉네임 설정으로 진행된다.

진입점

알림함·설정 진입 — 홈 헤더 & 마이 상단

알림함(벨)과 설정(톱니)은 홈 헤더와 마이 탭 상단 두 곳에 상시 노출된다. 리워드(교환소) 화면 상단에도 설정 톱니가 있다. 홈 벨 아이콘의 빨간 점은 현재 데모에서 정적 표시다(미읽음 수와 무관 — 하단 미결 참조).

홈 헤더 — 벨(알림함) + 톱니(설정)
마이 탭 상단 — 벨 + 톱니 동일 진입
3
알림함

전역 알림함 — 사용자 이벤트 기반 실데이터

알림함은 별도 알림 컬렉션 없이 사용자의 실제 이벤트 로그 4종(구매 포스트백 · 기프티콘 교환 · 친구초대 · 포인트 적립)을 통합해 시간순으로 렌더한다. 날짜 버킷(오늘/어제/이번 주/이전)으로 묶이며, 우측 상단에 '모두 읽음' 버튼이 있다. 아래 두 상태 모두 실제 계정으로 만든 화면이다.

신규 가입 직후

활동 없음 — 빈 상태

가입 직후 활동이 없으면 빈 상태 안내가 노출된다. 필터 탭 뱃지는 전부 0.

"알림이 없어요" 빈 상태
활동 발생 후

구매·출석·룰렛 후 자동 누적

구매 시뮬(67,000원) + 출석체크(+100P) + 룰렛 당첨(+200P)을 실제 수행한 직후의 알림함.

전체 3건 — 이벤트 2 + 구매 1
알림함 보관 기간·읽음 처리·뱃지 규칙은 마스터 정책서에 정의가 없다(FunctionalSpec 소관). 현재 데모는 이벤트 로그 파생형이라 별도 보관 정책 없이 원본 로그를 그대로 따른다.

알림함

필터 탭 — 전체 / 구매 / 티켓 / 이벤트

상단 필터 탭 4개가 실제 건수 뱃지와 함께 노출되고, 탭 전환 시 해당 유형만 필터링된다. 구매=포스트백, 티켓=기프티콘 교환, 이벤트=친구초대·포인트 적립(출석·룰렛 등) 유형이 매핑된다.

'구매' 탭 — 포스트백 알림만 필터
'이벤트' 탭 — 출석·룰렛 적립 알림만 필터
5
설정

설정 화면 — 알림 3종 스위치 + 5개 그룹

설정은 알림 / 홈 화면 / 표시 / 데이터 / 정보 5개 그룹이다. 최상단 알림 그룹에 정책 확정안대로 3종 스위치가 배치된다: 구매 알림(거래성, 기본 켜짐) · 이벤트-티켓 알림(룰렛·응모·당첨·만료 임박 D-30/D-7, 기본 켜짐) · 마케팅 정보 수신(기본 꺼짐). 각 스위치는 푸시·인앱에 공통 적용되는 1토글이다(1단계 단순화 — 채널 분리 없음).

알림 3종 스위치 (마케팅만 기본 OFF) + 홈 화면·표시 그룹
하단 — 다크모드(2차 예정) · 기본 정렬 · 캐시 삭제 · 약관 · 앱 버전
'홈 화면에 쿠팡 바로 가기 추가'는 실동작(cpkOpen() — 제휴 적립 유지 바로가기 안내). 다크모드는 화면에 '2차 업데이트 예정'으로 명시. 기본 정렬·캐시 삭제는 현재 데모에서 표시만 있는 상태(하단 미결 참조).
6
설정

스위치 변경 — 확정 정책 vs 현재 데모 동작

확정 정책: 스위치 상태는 서버에 저장되고 계정 기준으로 기기 간 동기화되며, 미연동 게스트는 로컬 저장 후 카카오 연동 시 승계된다. 현재 데모는 토글 UI와 변경 토스트까지만 구현되어 있고 서버 저장은 없다 — 화면 토스트에도 '(프로토타입)'으로 명시되어 있다.

마케팅 수신 토글 ON → "설정이 변경되었어요 (프로토타입)" 토스트
마케팅 동의는 변경분을 새로 기록해 이력을 보존한다(과거 동의 내역 보존 — DataModel 정책). 동의/철회 시점 기록이 법적 근거가 되므로 실구현 시 서버 저장과 함께 필수.
7
CMS

운영자: 알림 관리 — 상한 정책 · 작성 발송 · 발송 내역

CMS '알림 관리' 메뉴 한 곳에서 FCM 발송 상한 정책(거래 상한 없음 / 기타 일 2건 / 야간 22~08시 미발송)을 상시 확인하고, 채널(앱 푸시 / 카카오 알림톡 / 동시)·대상(전체·등급별·미접속·세그먼트)·발송 시점(즉시/예약)을 지정해 알림을 작성·발송한다. 발송 완료 건은 불변 로그로 남고 '예약' 상태만 취소·수정할 수 있다. 폼 하단에 "마케팅성 알림은 수신 동의 회원에게만 발송, 서비스 필수 알림은 동의 여부와 무관하게 발송" 원칙이 명시돼 있다.

CMS > 콘텐츠 > 알림 관리 — 발송 통계 · FCM 상한 정책 · 작성 폼 · 발송 내역
현재 데모의 알림 관리는 화면 구성까지만이고 발송 버튼은 토스트 처리(실발송·저장 없음)다. 인접 메뉴(문의·기프티콘 등)는 실연동인 것과 대비됨 — 하단 미결 참조.

운영자: 마케팅 수신 동의 — 회원 집계 · 약관 원문

회원 목록 상단에서 마케팅 수신 동의 집계(동의 수·비율)를 확인하고, 리스트에 '마케팅 수신' 컬럼과 '전체 마케팅' 필터가 있다. 약관 관리의 '마케팅 수신 동의' 탭에서 동의 항목(앱 푸시 / 카카오 알림톡)·발송 내용·수신 거부 방법 원문을 운영한다.

미결 · 확인 필요 (형님 확정 대기)

임의로 정하지 않고 표시만 한 항목. 확정되면 이 문서와 데모에 반영한다.

정책 지위알림 3종 스위치 정책의 SoT 정식 편입 여부. 마스터 정책서에서 해당 블록(DEVQA 2026-06-18 반영 패널)은 "검토용 반영 — 본 섹션 편입 여부는 형님 확정" 상태다. 이 스토리보드는 그 확정안을 기준으로 작성했으니, 편입 확정이 나야 정책 지위가 완결된다.
구현 갭설정 3종 스위치가 데모에서 저장되지 않는다. 정책은 서버 저장 + 기기 간 동기화 + 게스트 로컬 저장 후 승계인데, 현재 데모 토글은 UI 토글 + "(프로토타입)" 토스트뿐이고 Firestore 쓰기가 없다. 실구현 시 users 필드 설계(예: notif_pick / notif_event / marketing_consent)와 동의 이력 기록이 함께 필요하다.
구현 갭CMS 알림 관리가 목업 상태. 발송 버튼은 토스트만 출력(발송 큐 저장 없음), 발송 내역은 항상 빈 상태로 세팅된다. 데모 단계에서 실연동(발송 로그 저장·열람율 집계)까지 만들지 여부 확정 필요.
구현 갭설정의 '기본 정렬'·'캐시 삭제'가 무동작. 기본 정렬은 선택 UI 없음, 캐시 삭제는 12.4MB 하드코딩 표시에 핸들러 없음. 유지할 기능인지(웹 데모 특성상 캐시 삭제 개념이 모호) 정리 필요. 다크모드는 '2차 업데이트 예정'으로 화면에 명시되어 있어 미결 아님.
정책 공백알림함 운영 규칙 미정의. 보관 기간·읽음 처리·미읽음 뱃지 규칙이 마스터 정책서에 없다(FunctionalSpec 소관으로만 참조). 현재 데모는 미읽음 표시가 사실상 무동작이고, 홈 벨의 빨간 점도 정적 CSS라 실제 미읽음 수와 무관하다. 이벤트 로그 파생형 구조를 유지할지, 별도 알림 컬렉션으로 갈지도 개발 전 결정 필요.
문구 불일치약관 원문 "마이 > 알림 설정" vs 실제 화면 "설정". 마케팅 수신 동의 약관의 수신 거부 경로 문구가 실제 화면명과 다르고, "개별 항목별 수신 거부"라는 표현도 실제 구현(3종 통합 스위치)과 정합 확인이 필요하다.
UX 미정마케팅 동의/철회 시 안내 UX 미확정. 동의 이력 기록 정책은 있으나, 동의/철회 순간의 안내 문구(일시 고지 토스트 등, 정보통신망법 고지 관행)는 정해진 게 없다.

  • 알림함은 별도 컬렉션이 아니라 postbacks · gifticon_exchanges · invites · point_transactions 4개 이벤트 로그를 통합 렌더하는 파생형 구조다.
  • 알림 권한 프리퍼미션(1단계)은 온보딩 화면이지만 알림 기능의 시작점이라 본 스토리보드에 포함했다. 가입·로그인 전체 흐름은 별도 '가입·로그인' 스토리보드 소관.
  • 정책 수치의 단일 원본은 마스터 정책서(policyspec.html)이며, 본 문서는 그 발췌·시각화다. 충돌 시 정책서가 우선한다.
