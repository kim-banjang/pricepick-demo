1
유저

홈 미션 카드 — "행운 룰렛" 진입

홈 탭의 미션 카드 4종(쿠팡 구경 · 행운 룰렛 · 친구 초대 · 연속구매) 중 "행운 룰렛" 카드로 진입한다. 카드에는 "하루 1회" 배지가 노출된다. 출석(쿠팡 구경)과 무관한 독립 상시 이벤트다.

홈 미션 카드 · "행운 룰렛 — 하루 1회" (게스트 신규 계정, 0P)
2
유저

룰렛 메인 — CMS 설정 6칸을 그대로 렌더링

진입할 때마다 CMS가 저장한 settings/roulette 문서를 다시 읽어 6칸(유형·값)을 그대로 그린다. 하드코딩이 아니라 운영자가 바꾸면 앱이 바뀐다. 문서가 없으면 기본 시드값으로 대체. 상단 배지에 오늘 남은 기회(1회)가 표시된다.

칸123456
구성 (시드)이벤트 티켓 1장꽝포인트 100P이벤트 티켓 2장꽝포인트 200P
룰렛 초기 상태 · 6칸(포인트 100/200P · 이벤트 티켓 1/2장 · 꽝 2칸) + "돌리기"
위 표는 CMS settings/roulette에 저장된 현재 시드값이며, 8번 CMS 설정 화면의 저장값과 동일하다. 확정값이 아니라 형님 확정 대기 상태의 임시 운영값이다.
3
추첨 · 분기

돌리기 — 6칸 균등(1/6) 추첨, 결과 4종

"돌리기"를 누르면 약 4.4초 스핀 후 포인터가 가리키는 칸으로 결과가 확정된다. 6칸 균등 확률이며 칸별 가중치는 없다(가중치가 필요하면 CMS에서 같은 보상을 여러 칸에 배치). 결과는 화면 하단 결과 카드로 노출되고, 당첨 즉시 지급까지 완료된다. 이벤트 티켓 칸이라도 지급 한도(일 5장·월 30장)를 넘으면 미지급 처리된다.

당첨 · 포인트

포인트 칸 (100P / 200P)

당첨 즉시 포인트 지급. 결과 카드에 "축하해요! 보상이 지급됐어요"와 지급액 표시.

"100P 당첨" · 실제 지급 완료
당첨 · 이벤트 티켓

이벤트 티켓 칸 (1장 / 2장)

당첨 즉시 이벤트 티켓 발급(발급일 +100일 만료). 경품 응모 전용.

"이벤트 티켓 1장 당첨" · 실제 발급 완료
꽝

꽝 칸 (2칸)

지급 없음. "아쉽지만 다음 기회에 — 내일 다시 도전해보세요" 후 오늘 참여 종료.

"꽝 — 내일 다시 도전해보세요"
한도 초과 · 미지급

이벤트 티켓 칸 + 한도 소진

이벤트 티켓 칸에 멈춰도 일 5장·월 30장 한도 초과분은 지급되지 않는다.

"이벤트 티켓 지급 한도 초과 (일 5장 · 월 30장)"
한도 초과 화면은 실제 지급 로직으로 재현한 실캡처다 — 게스트 계정에 당일 이벤트 티켓 5장을 실제 적립(일 한도 소진)시킨 뒤 이벤트 티켓 칸에 멈추게 하자, 지급 함수가 한도를 확인하고 0장 미지급으로 처리하며 이 결과 카드를 띄웠다. 배분값·당첨 판정은 조작하지 않았고 미지급 판정만 실제 코드가 수행했다.
4
지급 반영

보상 즉시 반영 — 홈 · 혜택 탭

지급은 별도 대기 없이 즉시다. 포인트는 홈 상단 포인트 잔액에, 이벤트 티켓은 혜택 탭 "이벤트 티켓" 카드 수량에 바로 반영된다. 이벤트 티켓은 경품 응모 전용이라 적립(등급) 티켓과 완전히 분리돼 표시된다.

100P 당첨 직후 홈 · 0P → 100P 즉시 반영
이벤트 티켓 2장 당첨 직후 혜택 탭 · "이벤트 티켓 2장" 반영 (경품 응모 전용)
5
유저

활동 내역 — 당첨 보상 영구 기록

마이 > 활동 내역에 지급 건이 기록된다. 포인트는 "행운 룰렛 당첨 +100P"로, 이벤트 티켓은 "이벤트 티켓 적립 +1장"으로 남는다. 2장 당첨은 티켓 문서가 장 단위로 발급되므로 +1장 기록 2건으로 표시된다.

활동 내역 · "행운 룰렛 당첨 — 포인트 +100P"
활동 내역 · 2장 당첨 = "이벤트 티켓 적립 +1장" × 2건
6
유저

하루 1회 잠금 — 재진입 상태

참여 후 재진입하면 버튼이 "오늘 이미 참여했어요"로 비활성화되고, 상단 배지가 "오늘 참여 완료 · 내일 다시 오세요"로 바뀐다. 매일 자정에 초기화된다.

참여 완료 상태 · "오늘 이미 참여했어요" (버튼 비활성)
데모 구현 한계: 하루 1회 판정이 기기(브라우저) 로컬 저장 날짜(localStorage) 기준이라 서버 검증이 없다. 앱 데이터 삭제·기기 변경으로 우회 가능하므로 실서비스 개발 시 서버 기준 판정 필요. (개발팀 전달 사항)
7
운영자 · CMS

CMS "행운룰렛 설정" — 6칸 유형·값 편집 (실연동)

CMS > 티켓 · 보상 > 행운룰렛 설정에서 6칸 각각의 유형(꽝 / 이벤트티켓 / 포인트)과 값(장수 / P 액수)을 편집해 저장하면 settings/roulette에 기록되고, 앱은 다음 진입 시 그대로 반영한다. 꽝 선택 시 값 입력은 자동 비활성화된다. 캡처 시점 저장값은 시드와 동일한 6칸(이벤트티켓 1장 / 꽝 / 포인트 100 / 이벤트티켓 2장 / 꽝 / 포인트 200) — 형님 확정 전 임시 운영값이다.

CMS > 행운룰렛 설정 · 실제 Firestore 저장값 로드 상태 ("저장된 설정을 불러왔습니다.")
8
정책 · 규칙

정책 요약 (SoT: 마스터 정책서 policyspec.html — "행운룰렛 독립 정책")

  • 성격 — 출석(쿠팡 구경)과 무관한 독립 상시 이벤트. 구매 없이 참여 가능.
  • 참여 — 하루 1회, 매일 자정 초기화.
  • 보상 — 포인트 또는 이벤트 티켓을 확률 지급. 확률·배분은 정책상 미결이라 운영자가 CMS에서 직접 정한다(6칸 균등 1/6 추첨).
  • 6칸 배분값 형님 확정 대기 — 현재 시드: 포인트 100P·200P / 이벤트 티켓 1장·2장 / 꽝 2칸. CMS에 저장돼 배포본이 이 값으로 운영 중이나 확정값 아님.
  • 이벤트 티켓 성격 — 경품 응모(추첨) 전용, 등급(적립) 티켓과 완전 분리. 발급일 +100일 만료.
  • 지급 한도 — 이벤트 티켓 어뷰징 한도 일 5장 / 월 30장이 룰렛 지급에도 적용(전 경로 합산). 한도 초과 당첨 건은 미지급 처리(결과 카드에 안내).
  • 포인트 지급 — 즉시 지급·즉시 반영, 활동 내역 "행운 룰렛 당첨"으로 기록. 구매와 무관한 무상 보상이라 환불·환수와 무관.
  • 계정 병합 시 — 룰렛 등 이벤트성 무상 지급분은 닉네임 계정 → 카카오 계정 병합 시 중복 방지를 위해 합산에서 제외(마스터 정책서 회원 정책).
  • 데모 구현 한계(개발 전달) — 하루 1회 판정이 기기 로컬 기준(서버 미검증) / 칸별 가중 확률 미지원(균등 1/6만).

참고 사항

  • 모든 화면은 배포본 pricepick-demo.vercel.app/app(게스트 신규 계정) · /cms(firstadmin) 에 헤드리스 브라우저(390×844·dsf 2 / CMS 1440×900)로 접속해 실제 참여·지급·기록 확인까지 수행한 뒤 캡처했다.
  • 결과 4종(포인트 / 이벤트 티켓 / 꽝 / 한도 초과)은 하루 1회 제한 때문에 계정을 나눠 재현했으며, 각 결과의 지급·미지급·원장 기록은 앱의 실제 로직이 그대로 수행된 것이다. 한도 초과분은 계정에 당일 5장을 실제 적립해 한도를 소진시킨 상태에서 이벤트 티켓 칸에 멈추게 해 재현했다(미지급 판정은 실제 코드가 수행).
  • 룰렛 6칸 구성의 단일 원본은 CMS settings/roulette 문서다. 앱 화면·이 문서의 캡처 모두 해당 저장값(시드) 기준이며, 배분값은 형님 확정 대기 상태다.
  • 정책 근거는 마스터 정책서(policyspec.html) "참여 — 행운룰렛(독립 정책)" 및 정책 임계값 표(행운룰렛 / 이벤트 티켓 적립 한도) 조항이다.
