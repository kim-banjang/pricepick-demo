이 문서의 모든 화면 썸네일은 실제 배포본(pricepick-demo.vercel.app)에서 테스트 계정(SB검증C)을 만들고 CMS 수동 지급으로 티켓(브론즈 12·실버 4·골드 2)을 실제 세팅한 뒤 조작·캡처한 스크린샷입니다. 정책 기준은 마스터 정책서(policyspec.html, SoT)입니다.
구현 상태 주의 — 등급 교환 화면(UI)은 배포본에 존재하지만 교환 실행 로직은 아직 미구현입니다(하단 CTA는 실행 없이 이전 화면 복귀만, Firestore 기록 없음). 아래 5단계의 "실행·기록"은 정책서·데이터 구조 기준으로 정의된 동작이며 현재 앱에서 동작하지 않습니다.

1
유저

혜택 탭 — 내 티켓 카드에서 진입

혜택 탭 상단 '내 티켓' 카드에 등급별 보유 수량(브론즈/실버/골드)이 표시된다. '내 적립 티켓 자세히 보기 ›'를 누르면 적립 티켓 상세로 이동한다.

혜택 탭 · 내 티켓 카드 (브론즈 12 · 실버 4 · 골드 2 — CMS 수동 지급으로 세팅한 실데이터)
2
유저

내 적립 티켓 — 규칙 안내 + '티켓 바꾸러 가기'

'내 적립 티켓' 화면은 랜덤 티켓 활성화 목록과 등급 티켓 보유 현황을 보여주고, 하단 '티켓 등급 바꾸기' 섹션에서 교환 규칙(단일 등급 사용, 만료 임박분부터 자동 사용, 적립일 승계)과 비율을 안내한 뒤 '티켓 바꾸러 가기 ›'로 교환 화면에 진입시킨다.

내 적립 티켓 · 보유 현황 + 등급 바꾸기 안내/진입
3
유저

티켓 등급 교환 — 보유 현황 · 비율 · 방향 선택

교환 화면 상단에 등급별 보유 현황, 교환 비율(브론즈 10장 ⇄ 실버 1장 / 실버 2장 ⇄ 골드 1장), 사용 기한 처리 규칙(FIFO 자동 사용 · 적립일 승계 · 만료일 연장 없음)이 표시되고, 교환 방향을 버튼으로 선택한다.

등급 교환 화면 · 보유/비율/방향 선택 (실배포 화면)
정책 불일치(정정 대상) — 배포 화면은 양방향 4버튼(실버→브론즈, 골드→실버 하향 포함)에 "어느 방향으로 바꿔도 보유 가치는 같아요"라고 안내하지만, 마스터 정책서(SoT)는 상향만 허용·하향 불가로 확정돼 있다. 방향은 SoT 기준으로 이미 확정됐고, 앱 4버튼 UI·안내 문구가 이 기준에 맞게 정정돼야 한다(교환 실행 로직 자체가 미구현이라 화면 수정은 구현 시 함께 처리 — 아래 "구현 필요 항목" 참고).
4
유저

수량 입력 → 교환 후 보유 시뮬레이션

보낼 티켓 수량을 직접 입력하거나 퀵버튼(+10/+50/+100/최대/초기화)으로 조정하면, 받을 티켓 수와 '교환 후 보유 현황'(등급별 before → after)이 화면에서 미리 확인된다. 보유 초과 입력 시 '보유 부족' 경고를 띄우는 구조.

교환 수량 · 브론즈 10장 → 실버 1장 (현재 4장 → 교환 후 5장)
교환 후 보유 현황(12→2 / 4→5 / 2→2) + 하단 포인트 ↔ 티켓 교환 섹션
정식 기능 확정 — 화면 하단 '포인트 ↔ 티켓 교환'(1,000P→브론즈 1장, 10,000P→실버 1장, 20,000P→골드 1장, 역방향 포함)은 존치로 확정됐다. 근거: 정책서(SoT) 4. 포인트(P) — "포인트 ↔ 티켓 쌍방 교환(10P=1원 기준)"과 형님 확정(포인트 시스템 도입 확정). 실행 로직은 등급 교환과 함께 아직 미구현.
5
시스템 (미구현)

교환 실행 · 기록 — 정의된 동작 (현재 앱 미구현)

정책·데이터 구조 기준으로 교환 실행 시 일어나야 하는 처리는 다음과 같다. ① 보낼 등급의 활성 티켓을 만료 임박분(적립일 오래된 것)부터 FIFO로 used 처리, ② 받을 등급 티켓을 신규 생성하되 원본 중 가장 이른 적립일을 승계(만료일 연장 없음 — 등급 티켓 만료는 적립일+1년), ③ 티켓 원장(ticket_transactions)에 교환 기록 추가, ④ 활동내역에 '등급 전환'으로 노출, ⑤ 완료 후 취소 불가. 활동내역에는 '교환' 필터가 이미 준비돼 있어(아래 캡처) 실행이 구현되면 여기에 기록이 쌓인다.

마이 > 활동 내역 · '교환' 필터 — 등급 전환 기록이 남을 자리 (실행 미구현이라 현재 빈 상태) · "영구 조회 · 탈퇴 전까지 모든 이력 보존"
현재 배포본: 화면 하단 '티켓 교환하기' CTA는 실행 없이 이전 화면으로 복귀만 하며, 수량 스테퍼·시뮬레이션 숫자도 계산 로직 없이 고정값이다. 등급 교환용 Firestore 쓰기 함수 자체가 없다. 또한 정책상 "티켓 사용은 카카오 연동 후에만 가능(게스트는 적립만)"인데, 이 게이팅도 아직 앱에 없다.
6
운영자 · CMS

CMS — 교환 정책 패널(조회 전용) + 티켓 원장

등급 교환 전용 CMS 관리 화면은 없다(별도 운영 개입이 없는 자동 등가 전환이기 때문). CMS에서는 ① '운영 정책' 페이지의 '티켓 등급 간 교환 정책' 패널에서 비율·환산가치(브론즈 100원/실버 1,000원/골드 2,000원) 기준을 조회하고, ② '티켓 내역'(append-only 원장)에서 회원별 티켓 발행·사용·회수 기록을 확인하며, ③ 오류·CS 보정이 필요하면 '수동 지급 / 회수' 모달로 등급별 수량을 직접 지급·회수한다(사유 필수, 로그 기록, 되돌리기 불가). 교환 실행이 구현되면 이 원장에 교환 기록이 쌓이는 구조다.

CMS > 운영 정책 · 등급 간 교환 정책 패널 — SoT 확정은 상향만, 패널의 "양방향" 서술은 정정 필요(정책 변경 아님·표기 오류)
CMS > 티켓 내역 — 실 Firestore 원장(발행/사용/회수, 수동 지급/회수 포함) · 테스트 지급·교환 기록이 실제 반영된 화면
CMS > 티켓 내역 > 수동 티켓 지급 / 회수 — 대상 회원·처리 유형·등급별 스테퍼·사유(필수) · 실동작 (캡처는 처리 전 상태)

SoT 정정 반영 — 이번 대조로 확정된 항목. 스토리보드는 아래를 정본으로 서술한다.
  • 교환 방향 — SoT 기준 상향만 허용, 하향 불가로 이미 확정. 배포 앱 화면(방향 4버튼 + "어느 방향으로 바꿔도" 문구), CMS 운영 정책 패널("양방향 교환"), 구 스토리보드 인덱스 문구("전 방향 허용")는 전부 SoT와 어긋나는 오표기이며, 정책 재확인 대상이 아니라 화면·문구 정정 대상이다.
  • 사용 조건(연동 게이팅) — SoT 기준 티켓·포인트 사용(교환 포함)은 카카오 연동 후에만 가능(게스트는 적립만). 정책은 확정, 앱 구현만 미완 상태.
  • 포인트 ↔ 티켓 교환 — 존치 확정(형님 확정, 2026-07-09). 근거: SoT 4. 포인트(P) "포인트 ↔ 티켓 쌍방 교환"(10P=1원, 브 1,000P·실 10,000P·골 20,000P). 앱 화면의 해당 섹션은 정식 기능이며 더 이상 존치 여부를 재확인할 필요가 없다.

구현 필요 항목 (정책 전부 확정 — 형님 결정 불필요)

  1. 실행 로직 미구현 — 등급 교환·포인트↔티켓 교환 모두 앱·CMS에서 실행이 구현되지 않은 UI 목업 상태. 구현 우선순위만 남음.
  2. 연동 게이팅 미구현 — "사용은 카카오 연동 후" 정책이 앱 교환 흐름(등급 교환·포인트↔티켓 교환·기프티콘 교환)에 적용되지 않아 게스트도 사용 가능한 상태.
  3. 방향 제한 UI 미반영 — 앱 4버튼(양방향)·CMS 패널 서술을 SoT 기준 상향만으로 고치는 작업. 실행 로직 구현과 함께 처리.
